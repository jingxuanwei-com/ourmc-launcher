## 一、技术栈与框架

| 项 | 选型 | 说明 |
|---|---|---|
| 运行时 | .NET 8.0 (`net8.0-windows`) | 单一 Windows 桌面目标 |
| UI 框架 | Blazor WebView (`Microsoft.AspNetCore.Components.WebView.WindowsForms` 8.0.*) | 用 WebView2 渲染 Razor 组件 |
| 宿主容器 | Windows Forms | `LauncherForm` 自定义无边框窗口（圆角/深色/拖拽/淡入动画，调 dwmapi/user32） |
| 语言 | C# 12 / Razor | 启用 `Nullable` + `ImplicitUsings` |
| 前端 | 原生 HTML/CSS/JS | `index.html` 承载 Blazor，`liquid-glass.js` 毛玻璃，JS 桥接窗口控制/Toast |
| 依赖注入 | `Microsoft.Extensions.DependencyInjection` | 全部服务在 `Program.cs` 注册为 Singleton |
| 发布 | `dotnet publish -c Release -r win-x64 --self-contained -p:PublishSingleFile=true` | 单文件自包含 exe，免装运行时（`build-release.bat` / `build.ps1`） |
| 数据存储 | `%APPDATA%\.minecraft\` 为游戏目录，`%APPDATA%\.minecraft\oml\` 为启动器数据；设置在 `%LOCALAPPDATA%\OMLLauncher\settings.json` | 见 `PathService` |

**架构特点**：WinForms 只做窗口壳，所有 UI 逻辑在 Blazor Razor 组件里；C# 服务层负责全部业务（下载/启动/文件系统/网络），通过 DI 注入到组件；JS 仅做窗口拖拽、Toast、iframe 等少量桥接。

### 数据流

```
用户操作 → Razor 组件 → Service（DI 注入）→ 文件系统 / 网络 / 进程
              ↓ AppState.OnChange 事件
         StateHasChanged → UI 刷新
```

- `AppState` 是全局状态容器，属性 setter 触发 `OnChange`，`Main.razor` 订阅后刷新。
- 窗口操作走 JS→`[JSInvokable]`→`WindowService`（拖拽/最小化/关闭）。
- 游戏输出通过 `LaunchService` 重定向 → `LogService` 收集 → `LogViewerPage` 展示。

## 三、内核链路（下载 → 环境检查 → 启动）

这是启动器的核心链路，也是当前维护重点。

```
DownloadsPage → DownloadService.DownloadVersionAsync()
                  ├─ 版本清单 (version_manifest_v2.json)
                  ├─ 版本 JSON
                  ├─ client.jar（SHA1 校验）
                  ├─ 依赖库（libraries/）
                  ├─ 资源索引 + 资源对象（assets/objects/<2位hash>/<hash>，8路并发，增量跳过）
                  ├─ log4j 配置（assets/log_configs/）
                  └─ natives 提取（versions/<id>/natives/）
        ↓
HomePage → LaunchService.Launch()
             ├─ EnvironmentService.CheckEnvironment()（Java/内存/磁盘/文件/依赖）
             ├─ ResolveVersionChain()（解析 inheritsFrom 继承链，合并 mainClass/libraries/arguments）
             ├─ BuildLaunchArguments()
             │     ├─ 现代 arguments.game/jvm 解析 + ${var} 替换
             │     ├─ 旧版 minecraftArguments 回退
             │     ├─ OS 规则判断（windows/x86/版本正则，feature 规则排除）
             │     ├─ classpath（版本jar + 父版本jar + 库，Maven坐标回退）
             │     ├─ 离线 UUID（Java 兼容的 v3 MD5）
             │     └─ log4j 参数（文件存在时追加）
             └─ Process.Start() + 输出重定向
```

**内核关键能力已验证**：现代 `arguments` 解析、旧版 `minecraftArguments`、版本继承、离线 UUID、OS 规则、classpath、log4j、资源对象下载 —— 对 1.8.9 ~ 1.21.6 五个真实版本 JSON 跑通，0 未解析占位符。

## 四、各模块完善程度

图例：✅ 完成 ｜ 🟡 基本可用（有缺口）｜ 🟠 半成品 ｜ ⚪ 桩

### 4.1 内核 — 下载/启动/版本/环境/实例

| 模块 | 状态 | 说明 / 关键缺口 |
|---|---|---|
| **DownloadService** | ✅ | 完整下载流水线（client jar + 库 + 资源索引/对象 + log4j + natives）；所有官方文件按长度/SHA1 校验，使用 `.part` 临时文件后原子落盘，任一关键文件失败会终止安装。缺口：natives 仍为全量解压。 |
| **LaunchService** | ✅ | 现代参数解析 + 继承链 + 离线 UUID + OS 规则 + classpath + log4j。缺口：仅离线（token/uuid 走离线，正版接入待他人）。 |
| **VersionService** | ✅ | 扫描已装版本、类型推断（Forge/Fabric/Quilt/OptiFine）、JSON 持久化。缺口：`GameVersion` 元数据字段少。 |
| **InstanceService** | ✅ | 实例 CRUD + 导入导出(zip) + 备份。缺口：实例路径硬编码未走 PathService；导入不校验 zip。 |
| **EnvironmentService** | 🟡 | 检查框架完整（5 项 + 分级报告），能实际探测 Java 版本。**缺口较多**：`GetRequiredJavaVersion` 硬编码版本映射、不读版本 JSON 的 `javaVersion` 字段、**不覆盖 1.21+**（fallback 到 Java 8，错误）；`CalculateRequiredSpace` 硬编码 2048MB；语言文件路径错误（`assets/indexes/minecraft/lang`，实际在 `assets/objects`）；`CheckLibraries` 手推 Maven 路径、不用 `downloads.artifact.path`、不查 rules；`Is64BitJava` 靠路径含"64"不可靠。 |
| **VersionManagementService** | 🟠 | 框架在但完整性检查基本失效：`GetExpectedNativeFiles` 硬编码旧 DLL 名（对 LWJGL3 现代版全错）；`CleanupOldVersionsAsync` 空实现；`CheckLibraryRules` 用 `OSVersion.Platform`="Win32NT" 比对 "windows" → **永不匹配**；`CalculateVersionSize` 把整个 libraries 目录算进单版本。 |

### 4.3 模组子系统（完善度最低，建议重构）

四个服务功能高度重叠且各有根本性 bug，是全项目质量最差的区域。

| 模块 | 状态 | 关键缺口 |
|---|---|---|
| **ModService** | 🟡 | `GetModsDir` 忽略 versionName → **无版本隔离**（所有版本共享 mods/）；Forge `authorList`（数组）用 `GetString()` 读取会抛异常被吞；禁用机制非标准。 |
| **ModManagementService** | 🟠 | `ParseMcmodInfo` **把 JSON 当属性文件逐行 Split 解析** → 对 mcmod.info 完全失效；Fabric `authors`/`contact` 当字符串读会抛异常；`CheckModCompatibilityAsync` 恒返回 true。与 ModService 启停机制不同（`.disabled` 后缀 vs 移目录），共存混乱。 |
| **ModRepositoryService** | 🟠 | Modrinth search 用 **POST**（应为 GET，facets 格式也错）→ 大概率失败；`GetModrinthDetails` 读 project 不返回的 `members` 字段；CurseForge 是桩；**无模组下载方法**；`DetectConflicts` 仅文件名去重。 |
| **ModDependencyService** | 🟠 | `ParseFabricDependencies` 期望 `depends` 是**数组**（实际是对象）→ 标准 Fabric 模组依赖完全解析不出；Forge `mods.toml` 是桩；mcmod.info 同样当属性文件解析；注入了 `DownloadService` 却从不调用（"自动下载缺失依赖"未实现）。 |

## 六、已知关键缺陷清单（按影响排序）

**P0 — 影响启动成功**
1. `EnvironmentService.GetRequiredJavaVersion` 不读版本 JSON 的 `javaVersion`、不覆盖 1.21+ → 现代版本可能被误判 Java 版本而拦截启动。
2. `EnvironmentService.CheckLibraries` 不用 `downloads.artifact.path`、不查 rules → 误报库缺失，可能阻止启动。

**P1 — 功能失效**
3. `ModManagementService.ParseMcmodInfo` / `ModDependencyService` 把 JSON 当属性文件解析 → 模组元数据/依赖解析完全失效。
4. `ModRepositoryService.SearchModrinthMods` 用 POST（应为 GET）→ 在线搜模组大概率失败。
5. `PythonAutomationService` 脚本不存在 + 进程配置互斥 → 整个服务不可用。
6. `AIService` Anthropic 解析 bug + DeepSeek 死代码 → 两渠道失效。
7. `LaunchPresetService.GetSystemMemoryMB` ≈ 0 → 预设推荐不可信。

**P2 — 数据/体验**
8. `SkinService.GetUserProfileAsync` 恒返回 null。
9. `VersionManagementService` 完整性检查多处失效（DLL 名、规则匹配、大小计算）。
10. `PerformanceService.GenerateOptionsFile` `chatVis` 缺值 + 多项设置未写入。
11. `EnhancedDownloadService` 漏资源对象、进度恒 0（未被主流程使用）。
12. UI 多处"按钮只弹通知"（启动优化/启动参数/崩溃复制）。

**共性风险**：敏感信息（密码/ApiKey/Token）明文落盘；多服务无并发锁写同一文件；异常被 `catch{}` 静默吞；服务职责重叠（Download×2、Launch×2、Mod×4）。
